# Лабораторна робота №1: Розгортання Web-сервісу з автоматизацією

## Варіант індивідуального завдання
Розрахунок варіанту виконано для студента з порядковим номером у списку **N = 11**.

* $V_2 = (11 \% 2) + 1 = 2$ -> **База даних:** PostgreSQL. **Конфігурація:** через файл `/etc/mywebapp/config.json`.
* $V_3 = (11 \% 3) + 1 = 3$ -> **Тематика:** Simple Inventory (сервіс обліку обладнання).
* $V_5 = (11 \% 5) + 1 = 2$ -> **Порт застосунку:** 5200.

---

## Документація по розробленому веб-застосунку

### Призначення
**Simple Inventory** — це веб-застосунок для обліку обладнання. Він дозволяє зберігати інформацію про предмети інвентарю (назву та кількість), а також переглядати деталі по кожному запису. Застосунок написаний мовою Python з використанням мікрофреймворку Flask.

### Налаштування середовища (локально)
1. Встановіть Python 3 та PostgreSQL.
2. Створіть віртуальне середовище: `python3 -m venv venv` та активуйте його.
3. Встановіть залежності: `pip install -r requirements.txt`.
4. Створіть локальний файл `config.json` з реквізитами доступу до вашої локальної БД.
5. Запустіть міграцію: `python3 migrate.py`.

### Запуск застосунку
Для локального запуску виконайте: `python3 app.py`. Застосунок буде доступний за адресою `http://127.0.0.1:5200`.

## Документація по API-ендпоінтах

Застосунок підтримує Content-Negotiation. Якщо в заголовку запиту вказано `Accept: text/html`, дані повертаються у вигляді простої HTML-таблиці. За замовчуванням повертається JSON.

| Ендпоінт | Метод | Опис |
| :--- | :--- | :--- |
| `/health/alive` | GET | Перевірка стану застосунку (завжди повертає 200 OK). |
| `/health/ready` | GET | Перевірка підключення до БД (повертає 200 OK або 500 Error). |
| `/` | GET | Кореневий ендпоінт. Віддає виключно HTML зі списком усіх ендпоінтів бізнес-логіки. |
| `/items` | GET | Виводить список усіх предметів в інвентарі (`id`, `name`). |
| `/items` | POST | Створює новий запис. Тіло запиту: `{"name": "Назва", "quantity": 10}`. |
| `/items/<id>` | GET | Виводить детальну інформацію по запису (`id`, `name`, `quantity`, `created_at`). |

## Документація по розгортанню

### Базовий образ віртуальної машини
Для розгортання рекомендується використовувати офіційний образ **Ubuntu Server 22.04 LTS** або **24.04 LTS**.
* Завантажити можна з офіційного сайту: [ubuntu.com/download/server](https://ubuntu.com/download/server)
* Також підійде локальне середовище на базі WSL2 (Ubuntu).

### Вимоги до ресурсів ВМ
* **CPU:** мінімум 1 ядро.
* **RAM:** мінімум 1 ГБ (рекомендовано 2 ГБ для стабільної роботи PostgreSQL).
* **Disk:** 10 ГБ вільного простору.
* Спеціальна розмітка диска не вимагається.

### Вхід на ВМ
Після встановлення ОС на віртуальну машину, підключіться до неї через консоль гіпервізора або по SSH, використовуючи обліковий запис, створений під час інсталяції Ubuntu (цей користувач повинен мати права `sudo`). 

*Примітка: після виконання автоматизації цей дефолтний користувач буде заблокований згідно з вимогами.*

### Як запустити автоматизацію
1. Клонуйте цей репозиторій на віртуальну машину: 
   `git clone <посилання_на_ваш_репозиторій>`
2. Перейдіть у папку проєкту: `cd devops-lab1-mywebapp`
3. Надайте скрипту права на виконання: `chmod +x setup.sh`
4. Запустіть скрипт від імені суперкористувача: `sudo ./setup.sh`

### Інструкція з тестування розгорнутої системи
Після успішного виконання скрипта `setup.sh`:

1. **Перевірка Nginx:** Відкрийте браузер (або використайте `curl`) і перейдіть за IP-адресою вашої віртуальної машини: `http://<IP_адреса_ВМ>/`. Ви маєте побачити HTML-сторінку зі списком ендпоінтів.
2. **Перевірка БД та сервісу:** Виконайте запит `curl http://<IP_адреса_ВМ>/health/ready`. Відповідь має бути `OK`.
3. **Перевірка бізнес-логіки:** * Створіть запис: `curl -X POST -H "Content-Type: application/json" -d '{"name":"Laptop", "quantity":5}' http://<IP_адреса_ВМ>/items`
   * Отримайте запис у форматі JSON: `curl -H "Accept: application/json" http://<IP_адреса_ВМ>/items`
   * Отримайте запис у форматі HTML: `curl -H "Accept: text/html" http://<IP_адреса_ВМ>/items`
4. **Перевірка користувачів:** Спробуйте зайти по SSH під користувачами `student`, `teacher` та `operator` з паролями, вказаними у скрипті. Перевірте, що `operator` може лише керувати сервісами через `sudo systemctl`.
